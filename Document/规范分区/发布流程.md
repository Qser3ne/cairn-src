# 发布流程

## 目标

记录 Cairn 项目发布版本时需要同步的文件、验证命令和 Git 操作，避免运行时版本、打包版本与发布 tag 不一致。

## 版本来源

- `cairn/pyproject.toml` 的 `[project].version` 是 Python 包版本。
- `cairn/src/cairn/__init__.py` 的 `__version__` 是服务端 `/openapi.json` 等运行时元信息使用的版本。
- `cairn/uv.lock` 中 `name = "cairn"` 的 package version 需要与项目版本保持一致。

## 发布步骤

1. 同步更新上述三个版本位置。
2. 运行测试：

   ```bash
   PYTHONPATH=src /tmp/cairn-test-venv/bin/python -m pytest -q -s --capture=no tests
   ```

3. 提交 release commit，例如：

   ```bash
   git commit -m "chore: release v1.3"
   ```

4. 创建并推送版本 tag，例如：

   ```bash
   git tag v1.3
   git push origin main
   git push origin v1.3
   ```

## 本机生产部署

开发目录为 `/home/qser3ne/Application/carin-dev`，生产目录为 `/home/qser3ne/Application/carin`。日常开发完成后，使用仓库内脚本把已提交的 `main` 分支同步到生产目录并重启生产服务：

```bash
./scripts/deploy.sh
```

可选参数：

```bash
./scripts/deploy.sh --skip-tests
./scripts/deploy.sh --no-backup
```

部署脚本执行顺序：

1. 校验开发目录和生产目录都是 Git 仓库，且 `origin` 地址一致。
2. 校验两个目录都在 `main` 分支。
3. 拒绝部署开发目录或生产目录中未提交的 tracked 文件改动；`.venv-test/` 等未跟踪目录不会阻塞部署。
4. 默认在开发目录的 `cairn/` 子项目运行 `uv run --group dev pytest -s`，除非显式传入 `--skip-tests`。
5. 将开发目录的 `main` 推送到 `origin`。
6. 在生产目录执行 `git fetch origin` 和 `git pull --ff-only origin main`。
7. 默认把生产目录的 `datas/` 备份到 `datas.backup/<timestamp>/`，除非显式传入 `--no-backup`。
8. 调用生产目录已有的 `./start.sh` 重建主服务镜像、重建 worker 镜像、清理旧动态 worker 容器并重启 dispatcher。
9. 通过 `curl -f http://127.0.0.1:8000/projects` 做服务健康检查。

部署保护项：

- 脚本不会自动 `git add`、`git commit` 或 `git reset --hard`。
- 脚本不会覆盖生产本地的 `dispatch.yaml`、`datas/`、`.venv-test/`。
- 生产目录如存在已跟踪文件修改，脚本会停止，先人工确认这些改动应提交、暂存还是放弃。

## 公开仓库前检查

- README 必须明确说明 `Qser3ne/cairn-src` 是 `oritera/Cairn` 的 modified version，并基于当前 SRC-only 项目状态描述运行方式。
- README 不应包含 `personal and educational use`、商业授权联系、贡献双授权等下游无权代表上游承诺的表述。
- `LICENSE` 保持标准 GNU AGPLv3 正文；fork、版权归属和 attribution 说明放在 README。
- 推送到 `origin main` 后再执行 GitHub 可见性切换：

  ```bash
  gh repo edit Qser3ne/cairn-src --visibility public --accept-visibility-change-consequences
  ```
